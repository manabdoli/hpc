HPC Cluster: Getting Started Guide

A Practical Tutorial for New Users

Author

Mansour Abdoli

Published

March 13, 2026

1 Overview

This guide walks you through every step needed to access and use the HPC (High-Performance Computing) cluster — from requesting an account to running your first parallel job in MATLAB, R, or Python.

Note

Commands shown in this tutorial use <angle_brackets> to indicate values you must replace with your own (e.g., replace <your_username> with your actual university NetID).

2 Creating an Account

Account creation requires a request to IT. You cannot self-register.

Steps:

  1. Email the HPC support helpdesk at hpc-support@university.edu
  2. Include in your request:
    • Your full name and university NetID
    • Your department and PI (supervisor) name
    • A brief description of your intended use (e.g., “running Python ML jobs for my thesis”)
    • Estimated storage needs (if known)
  3. IT will provision your account within 2–5 business days and email you with:
    • Your cluster hostname (e.g., hpc.university.edu)
    • Your default storage quota
    • A link to the cluster usage policy (please read it!)
Important

Your HPC account is separate from your regular university login. You will authenticate using SSH keys (see next section) rather than a password.

3 Setting Up SSH Keys

SSH keys replace passwords with a cryptographic key pair: a private key (stays on your computer, never share it) and a public key (uploaded to the cluster).

3.1 Generate an SSH Key Pair

  • Mac / Linux
  • Windows (PowerShell)

Open Terminal and run:

ssh-keygen -t ed25519 -C "<your_email@university.edu>"

When prompted:

  • File location: Press Enter to accept the default (~/.ssh/id_ed25519)
  • Passphrase: Enter a strong passphrase (recommended) — this protects your private key

This creates two files:

File Description
~/.ssh/id_ed25519 Your private key — keep this secret
~/.ssh/id_ed25519.pub Your public key — this gets shared

Open PowerShell and run:

ssh-keygen -t ed25519 -C "<your_email@university.edu>"

Accept the default location (C:\Users\<you>\.ssh\id_ed25519) and set a passphrase.

Tip

If ssh-keygen is not found, enable the OpenSSH client:
Settings → Apps → Optional Features → Add a feature → OpenSSH Client

3.2 Copy Your Public Key to the Cluster

  • Mac / Linux
  • Windows (PowerShell)
ssh-copy-id -i ~/.ssh/id_ed25519.pub <your_username>@hpc.university.edu

If ssh-copy-id is unavailable, do it manually:

cat ~/.ssh/id_ed25519.pub | ssh <your_username>@hpc.university.edu \
  "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh <your_username>@hpc.university.edu `
  "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"

3.3 Test Your Connection

ssh <your_username>@hpc.university.edu

If successful, you will see the cluster’s welcome banner and a shell prompt like:

[<your_username>@login01 ~]$

4 Terminal-Based Access

4.1 Getting Connected

4.1.1 Mac / Linux

Mac and Linux have a built-in SSH client in the Terminal app (or iTerm2, Alacritty, etc.).

ssh <your_username>@hpc.university.edu

Optional: Create an SSH config alias so you can just type ssh hpc:

# Add to ~/.ssh/config (create the file if it doesn't exist)
Host hpc
    HostName hpc.university.edu
    User <your_username>
    IdentityFile ~/.ssh/id_ed25519
    ServerAliveInterval 60

Then connect with simply:

ssh hpc

4.1.2 Windows

Option 1 — Built-in OpenSSH (Windows 10/11, recommended):

Open PowerShell or Command Prompt:

ssh <your_username>@hpc.university.edu

Option 2 — PuTTY:

  1. Download PuTTY from putty.org
  2. In the PuTTY configuration:
    • Host Name: hpc.university.edu
    • Connection → SSH → Auth → Credentials: Browse to your .ppk private key
      (Use PuTTYgen to convert your id_ed25519 file to .ppk format)
  3. Click Open
Tip

Save your PuTTY session so you don’t have to re-enter settings each time.

4.2 Transferring Data via Terminal

4.2.1 Mac / Linux — Using scp

Copy a single file to the cluster:

scp /path/to/local/file.txt <your_username>@hpc.university.edu:~/data/

Copy a directory to the cluster (recursive):

scp -r /path/to/local/folder/ <your_username>@hpc.university.edu:~/data/

Copy a file from the cluster to your local machine:

scp <your_username>@hpc.university.edu:~/results/output.csv ~/Downloads/

4.2.2 Mac / Linux — Using rsync (recommended for large transfers)

rsync is faster than scp for large or repeated transfers — it only copies changed files.

# Sync local folder to cluster
rsync -avz --progress /path/to/local/folder/ <your_username>@hpc.university.edu:~/data/

# Sync cluster results back to local machine
rsync -avz --progress <your_username>@hpc.university.edu:~/results/ ~/local_results/
Flag Meaning
-a Archive mode (preserves permissions, timestamps)
-v Verbose output
-z Compress during transfer
--progress Show transfer progress

4.2.3 Windows — Using scp (PowerShell)

# Upload a file
scp C:\Users\<you>\data\file.txt <your_username>@hpc.university.edu:~/data/

# Download a file
scp <your_username>@hpc.university.edu:~/results/output.csv C:\Users\<you>\Downloads\

4.3 Scheduling Jobs with SLURM

The cluster uses SLURM (Simple Linux Utility for Resource Management) to schedule and manage jobs. You do not run computations directly on the login node — you submit job scripts that SLURM queues and executes on compute nodes.

4.3.1 Key SLURM Commands

Command Description
sbatch job.sh Submit a job script
squeue -u <your_username> View your queued/running jobs
scancel <job_id> Cancel a job
sinfo View available partitions and node status
sacct -j <job_id> View accounting info for a completed job
seff <job_id> View efficiency report for a completed job

4.3.2 SLURM Script Structure

Every SLURM job script is a Bash script with #SBATCH directives at the top:

#!/bin/bash
#SBATCH --job-name=my_job          # Job name (shows in squeue)
#SBATCH --partition=general        # Partition/queue to submit to
#SBATCH --nodes=1                  # Number of nodes
#SBATCH --ntasks=1                 # Number of tasks (processes)
#SBATCH --cpus-per-task=4          # CPU cores per task
#SBATCH --mem=8G                   # Total memory (e.g., 8 GB)
#SBATCH --time=02:00:00            # Wall-clock time limit (HH:MM:SS)
#SBATCH --output=logs/%j_out.txt   # Standard output file (%j = job ID)
#SBATCH --error=logs/%j_err.txt    # Standard error file
#SBATCH --mail-type=END,FAIL       # Email on job end or failure
#SBATCH --mail-user=<your_email>   # Your email address

# --- Your commands below ---
Warning

Always create the logs/ directory before submitting: mkdir -p logs

4.3.3 Running MATLAB Jobs

Interactive test (before submitting):

# Start an interactive session on a compute node
srun --ntasks=1 --cpus-per-task=2 --mem=4G --time=00:30:00 --pty bash

# Load MATLAB module
module load matlab

# Launch MATLAB (no display)
matlab -nodisplay -nosplash

Batch job script — run_matlab.sh:

#!/bin/bash
#SBATCH --job-name=matlab_job
#SBATCH --partition=general
#SBATCH --nodes=1
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=8
#SBATCH --mem=16G
#SBATCH --time=04:00:00
#SBATCH --output=logs/%j_out.txt
#SBATCH --error=logs/%j_err.txt

# Load MATLAB
module load matlab

# Run MATLAB script (no GUI)
matlab -nodisplay -nosplash -r "run('my_script.m'); exit"

Submit the job:

mkdir -p logs
sbatch run_matlab.sh

Parallel MATLAB with parpool — in your .m script:

% Use the number of CPUs requested in #SBATCH --cpus-per-task
parpool('local', str2num(getenv('SLURM_CPUS_PER_TASK')));

parfor i = 1:100
    result(i) = heavy_computation(i);
end

delete(gcp);

4.3.4 Running R Jobs

Batch job script — run_r.sh:

#!/bin/bash
#SBATCH --job-name=r_job
#SBATCH --partition=general
#SBATCH --nodes=1
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=4
#SBATCH --mem=8G
#SBATCH --time=02:00:00
#SBATCH --output=logs/%j_out.txt
#SBATCH --error=logs/%j_err.txt

# Load R module
module load R

# Run R script
Rscript my_analysis.R

Parallel R with parallel package — in your .R script:

library(parallel)

# Detect cores allocated by SLURM
n_cores <- as.integer(Sys.getenv("SLURM_CPUS_PER_TASK", unset = 1))
cat("Using", n_cores, "cores\n")

cl <- makeCluster(n_cores)

results <- parLapply(cl, 1:100, function(i) {
  heavy_computation(i)
})

stopCluster(cl)

Installing R packages on the cluster:

# Start R interactively
module load R
R

# Inside R, install to your personal library
install.packages("ggplot2", lib = "~/R/library")
Tip

Add export R_LIBS_USER=~/R/library to your ~/.bashrc so R always finds your personal packages.

4.3.5 Running Python Jobs

Recommended: use a Conda environment for managing packages.

Set up Conda environment (one-time):

module load anaconda   # or: module load miniconda

# Create environment
conda create -n myenv python=3.11 numpy pandas scikit-learn -y

# Activate
conda activate myenv

Batch job script — run_python.sh:

#!/bin/bash
#SBATCH --job-name=python_job
#SBATCH --partition=general
#SBATCH --nodes=1
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=8
#SBATCH --mem=16G
#SBATCH --time=06:00:00
#SBATCH --output=logs/%j_out.txt
#SBATCH --error=logs/%j_err.txt

# Load Anaconda and activate environment
module load anaconda
conda activate myenv

# Run Python script
python my_script.py

GPU job — add these directives for GPU-enabled partitions:

#SBATCH --partition=gpu
#SBATCH --gres=gpu:1               # Request 1 GPU
#SBATCH --cpus-per-task=4
#SBATCH --mem=32G

Parallel Python with multiprocessing:

import multiprocessing
import os

n_cores = int(os.environ.get("SLURM_CPUS_PER_TASK", 1))
print(f"Using {n_cores} cores")

def heavy_computation(i):
    # your work here
    return i ** 2

with multiprocessing.Pool(n_cores) as pool:
    results = pool.map(heavy_computation, range(100))

4.3.6 Job Arrays (Running Many Similar Jobs)

Job arrays let you run the same script hundreds of times with different inputs — ideal for parameter sweeps.

#!/bin/bash
#SBATCH --job-name=array_job
#SBATCH --array=1-50              # Run 50 tasks (indices 1–50)
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=2
#SBATCH --mem=4G
#SBATCH --time=01:00:00
#SBATCH --output=logs/%A_%a_out.txt   # %A = array job ID, %a = task index

module load python
conda activate myenv

# Use $SLURM_ARRAY_TASK_ID to select input
python my_script.py --input data/sample_${SLURM_ARRAY_TASK_ID}.csv

5 Connecting via Apps (GUI Clients)

GUI-based SSH clients make it easier to browse files and open a terminal without memorising commands.

5.1 Mac

5.1.1 Terminal (built-in)

Terminal.app is sufficient for most users. For a better experience, install iTerm2 from iterm2.com — it supports split panes, profiles, and session persistence.

5.1.2 Visual Studio Code (Recommended)

VS Code’s Remote - SSH extension lets you edit files on the cluster as if they were local.

  1. Install VS Code
  2. Install the Remote - SSH extension (Cmd+Shift+X → search “Remote - SSH”)
  3. Press Cmd+Shift+P → Remote-SSH: Connect to Host → <your_username>@hpc.university.edu
  4. VS Code will install a server on the cluster automatically

You can now open folders, edit files, and open an integrated terminal — all on the cluster.

5.2 Linux

5.2.1 Terminal (built-in)

Any terminal emulator works. Use the SSH command directly:

ssh <your_username>@hpc.university.edu

5.2.2 Visual Studio Code

Same Remote - SSH setup as Mac above. Install VS Code from code.visualstudio.com or via your package manager:

# Ubuntu/Debian
sudo snap install code --classic

5.3 Windows

5.3.1 Windows Terminal + OpenSSH (Recommended)

Install Windows Terminal from the Microsoft Store for a much better terminal experience. Then connect using:

ssh <your_username>@hpc.university.edu

5.3.2 PuTTY

A classic Windows SSH client. Download from putty.org.

  1. Host Name: hpc.university.edu
  2. Connection → SSH → Auth → Credentials: load your .ppk private key
  3. Save the session for future use

5.3.3 MobaXterm (Windows — highly recommended)

MobaXterm combines SSH, file transfer (SFTP), and X11 forwarding (for GUI apps) in one tool.

  1. Download MobaXterm Home Edition (free)
  2. Click Session → SSH
  3. Fill in Remote Host, username, and your private key path
  4. MobaXterm automatically opens an SFTP file browser alongside your terminal

5.3.4 Visual Studio Code

Same Remote - SSH workflow as Mac/Linux. Works identically on Windows.

6 Transferring Data via Apps (GUI)

6.1 Mac

6.1.1 Cyberduck

Cyberduck is a free, user-friendly SFTP client for Mac.

  1. Click Open Connection
  2. Select SFTP (SSH File Transfer Protocol)
  3. Server: hpc.university.edu
  4. Username: your cluster username
  5. SSH Private Key: browse to ~/.ssh/id_ed25519
  6. Click Connect

You can now drag-and-drop files between your Mac and the cluster.

6.1.2 Filezilla

FileZilla is a cross-platform SFTP client.

  1. Open File → Site Manager → New Site
  2. Protocol: SFTP
  3. Host: hpc.university.edu
  4. Logon Type: Key file → browse to ~/.ssh/id_ed25519
  5. Click Connect

6.1.3 Visual Studio Code (Remote - SSH)

Once connected via Remote - SSH, you can right-click any file in the Explorer panel and choose Download or drag files between the VS Code Explorer and your local Finder.

6.2 Linux

6.2.1 Filezilla

Same setup as Mac above. Install via:

sudo apt install filezilla   # Ubuntu/Debian
sudo dnf install filezilla   # Fedora

6.2.2 Thunar / Nautilus (File Manager SFTP)

Most Linux file managers support SFTP directly:

  • Nautilus (GNOME): press Ctrl+L, type sftp://<your_username>@hpc.university.edu
  • Thunar (XFCE): same — press Ctrl+L and enter the SFTP URL

6.3 Windows

6.3.1 MobaXterm

MobaXterm automatically opens an SFTP panel on the left when you connect via SSH. You can drag-and-drop files between the panel and Windows Explorer.

6.3.2 WinSCP

WinSCP is a dedicated SFTP/SCP client for Windows.

  1. File Protocol: SFTP
  2. Host name: hpc.university.edu
  3. User name: your cluster username
  4. Click Advanced → SSH → Authentication → load your private key (.ppk format via PuTTYgen)
  5. Click Login

6.3.3 Cyberduck (Windows)

Cyberduck also works on Windows — same setup as the Mac instructions above.

6.3.4 Filezilla (Windows)

Same setup as Mac/Linux. Download the Windows installer from filezilla-project.org.

7 Storage and Quotas

Most HPC systems provide several storage locations with different purposes and performance characteristics:

Location Path Purpose Backed Up? Notes
Home ~/ or /home/<user> Scripts, code, small files Yes Quota applies (e.g., 50 GB)
Scratch /scratch/<user> Large temporary data No High-speed; purged periodically
Project /project/<group> Shared group storage Yes Ask PI for access
Warning

Do not store large datasets in your home directory. Use /scratch for job I/O and move important results to /project or your local machine when done.

Check your current usage:

# Home directory quota
quota -s

# Scratch usage
du -sh /scratch/<your_username>

8 Environment Modules

The cluster uses modules to manage software versions. You must load a module before using software.

# List all available modules
module avail

# Search for a specific software
module avail python
module avail matlab

# Load a module
module load python/3.11
module load matlab/R2024a

# See currently loaded modules
module list

# Unload a module
module unload python/3.11

# Unload all modules
module purge
Tip

Add frequently used module load commands to your ~/.bashrc so they load automatically at login:

echo "module load anaconda" >> ~/.bashrc

9 Monitoring and Debugging Jobs

9.1 Check Job Status

# Your current jobs
squeue -u <your_username>

# All jobs on cluster
squeue

# Detailed info on a specific job
scontrol show job <job_id>

9.2 View Job Output in Real Time

# Follow output as it writes (like tail -f)
tail -f logs/<job_id>_out.txt

9.3 Job Efficiency After Completion

# Summary of CPU, memory, and time usage
seff <job_id>

# Detailed accounting
sacct -j <job_id> --format=JobID,JobName,State,Elapsed,CPUTime,MaxRSS
Tip

Optimise your resource requests. If seff shows you used only 20% of requested memory, reduce --mem in future jobs. Over-requesting resources means longer queue waits.

9.4 Common Issues

Problem Likely Cause Fix
Job stays in PENDING Waiting for resources Check squeue reason column; try a smaller resource request
module: command not found Module system not loaded Source your .bashrc: source ~/.bashrc
Out of memory error --mem too low Increase --mem and resubmit
Job cancelled after time limit --time too short Profile locally first; increase time limit
Permission denied on SSH Wrong key or username Verify ~/.ssh/authorized_keys on cluster

10 Best Practices

  1. Always test on a small dataset before submitting a full job. Use an interactive session (srun --pty bash) to verify your script works.

  2. Use scratch for I/O-intensive jobs. Reading/writing to /scratch is much faster than your home directory.

  3. Set realistic resource requests. Overestimate slightly, but don’t request 32 cores if your job won’t use them — this wastes cluster resources and increases your queue wait time.

  4. Clean up scratch regularly. Scratch space is not backed up and is often auto-purged after 30–90 days.

  5. Version-control your scripts. Keep your job scripts and analysis code in a Git repository.

  6. Check the cluster news page or mailing list before submitting large jobs — maintenance windows and node outages are announced there.
